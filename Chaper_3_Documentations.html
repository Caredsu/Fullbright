3.1 Developed System Overview

3.1.1 System Architecture

The Teacher Evaluation System is a comprehensive multi-platform solution designed to facilitate efficient teacher performance evaluation. The system comprises three primary user-facing applications (React Web PWA, Flutter Mobile, and React Admin Dashboard) supported by a robust Node.js/Express backend with MongoDB database.

3.1.4 Deployment Architecture

3.1.5 Key System Features

  • Multi-platform Support - Web (PWA), Mobile (Flutter), and Admin dashboard
  • Secure Authentication - JWT tokens for stateless authentication, no passwords for students
  • Offline-first Mobile - Queue submissions when offline, auto-sync when connection restored
  • Real-time Updates - Socket.io integration for live notifications in admin dashboard
  • Duplicate Prevention - Cross-device checks prevent re-evaluation of same teacher
  • Sequential Evaluation - 5-set form with progressive unlocking based on completion
  • Draft Management - Auto-save and recovery of incomplete evaluations
  • Admin Controls - System-wide evaluation enable/disable, user management, analytics
  • Data Privacy - Privacy consent modals before first evaluation
  • Responsive Design - Works on phones, tablets, and desktops

3.3 Data Structure and Implementations

3.3.1 MongoDB Collections and Schema

Collection: evaluations

Stores all student evaluations of teachers

{ "_id": ObjectId, "teacher_id": ObjectId, "student_id": String, "rating": Number (1-5), "feedback": String, "positive_feedback": String, "negative_feedback": String, "answers": { "question_id_1": "5", "question_id_2": "4", "question_id_3": "Great teacher" }, "questions_responses": Array, "created_at": ISODate, "updated_at": ISODate }

Collection: questions

Stores evaluation form questions organized into 5 sets

{ "_id": ObjectId, "text": String, "type": String (rating|feedback|text), "category": String, "set_number": Number (1-5), "choice_descriptions": { "1": "Strongly Disagree", "2": "Disagree", "3": "Neutral", "4": "Agree", "5": "Strongly Agree" }, "required": Boolean, "status": String (active|inactive), "display_order": Number, "created_at": ISODate, "updated_at": ISODate }

Collection: teachers

Stores teacher information

{ "_id": ObjectId, "first_name": String, "last_name": String, "middle_name": String, "email": String, "department": String (ECT|EDUC|CCJE|BHT), "status": String (active|inactive), "picture": String (Cloudinary URL), "picture_public_id": String, "created_at": ISODate, "updated_at": ISODate }

Collection: admins

Stores admin/staff user accounts

{ "_id": ObjectId, "username": String (unique), "email": String, "password": String (bcrypt hashed), "role": String (super_admin|admin|staff), "status": String (active|inactive), "created_by": ObjectId, "created_at": ISODate, "updated_at": ISODate }

Collection: sessions

Express session store for tracking active sessions

{ "_id": String, "session": Object (contains userId, token, role), "expires": ISODate (TTL index: 24 hours) }

Collection: settings

System configuration and toggles

{ "_id": "system", "eval_enabled": Boolean, "updated_at": ISODate, "updated_by": ObjectId }

3.3.3 Database Relationships

Relationship Type Description
Teachers ↔ Evaluations One-to-Many One teacher can have many evaluations via teacher_id foreign key
Questions ↔ Answers One-to-Many Evaluations store answers by question_id in answers object
Questions ↔ Sets One-to-Many Questions grouped by set_number (1-5), max 10 per set
Admins ↔ Sessions One-to-Many Admin sessions stored with session IDs linking to user accounts

3.3.4 Key Constraints and Validation Rules

Field Constraint Validation Rule
student_id (evaluations) Required, Unique per teacher Never 'anonymous' to prevent duplicate prevention bypass
rating (evaluations) Required Integer between 1-5 inclusive
teacher_id Required, Foreign Key Must reference existing teacher in teachers collection
set_number (questions) Required Integer 1-5, grouped into sequential sets
role (admins) Required Enum: super_admin, admin, or staff
eval_enabled (settings) Required Boolean toggle, checked before allowing evaluations

3.4 Implemented System Modules and Features

3.4.1 React Web Platform (Student Evaluation)

Core Pages

Page Route Purpose Key Features
Landing / Authentication entry point Student number login, data privacy consent modal, form validation
Dashboard /dashboard Teacher list and evaluation hub Search, department filter, recent evaluations, draft recovery, real-time notifications
Evaluation /evaluation/:teacher_id Multi-set evaluation form 5 question sets, sequential unlocking, auto-save draft, already-evaluated prevention

Core Components

  • ProtectedRoute - JWT token validation, redirects to login if unauthorized
  • AlreadyEvaluatedModal - Prevents duplicate submissions, shows warning dialog
  • DataPrivacyModal - Privacy consent with scrollable terms and acceptance checkbox
  • DraftRecovery - Auto-restores incomplete evaluations from browser storage
  • SessionTimeoutManager - 15-minute inactivity timer with warning countdown
  • InstallPrompt - PWA installation prompt for mobile browsers
  • Toast - Success/error notifications with auto-dismiss
  • ErrorBoundary - Catches React errors and displays fallback UI

Services and Hooks

  • api.js - Central axios client with JWT refresh token logic
  • useOfflineDetection - Detects online/offline state with "back online" notifications
  • useSessionTimeout - Tracks inactivity, auto-logout, warning countdown
  • useSessionManager - Session persistence and restoration

3.4.2 Flutter Mobile Platform (Cross-platform)

Core Screens

Screen Purpose Key Features
SplashScreen App initialization and bootstrapping Loading animation, data validation, route to next screen
DataPrivacyScreen Privacy policy display Scrollable terms, acceptance checkbox, legal compliance
StudentLoginScreen Student authentication Student number input, form validation, error handling
TeacherListScreen Teacher directory and entry point Department filtering, search, real-time status, offline support
EvaluationScreen Multi-set evaluation form 5 sets, sequential unlock, draft auto-save, offline queuing, error recovery
SuccessScreen Submission confirmation Success animation, auto-redirect, evaluation history
SettingsScreen User preferences Logout, device info, app settings

Core Services

  • api_service.dart - HTTP client with retry logic (3 attempts), internet check, exponential backoff
  • auth_service.dart - JWT token management, local auth state persistence
  • draft_service.dart - Local draft storage with expiration and progress tracking
  • duplicate_prevention_service.dart - Cross-device duplicate check with local fallback
  • offline_queue_service.dart - Queue submissions offline, retry (3 max), sync when online
  • connectivity_monitor_service.dart - Background connectivity monitor, auto-sync, 30-second check interval
  • evaluation_history_service.dart - Track evaluated teachers locally

Key Mobile Features

  • Offline Submission Queue - Queue evaluations when network unavailable, auto-sync when online
  • Background Connectivity Monitor - Continuous 30-second interval checks, auto-sync trigger
  • Retry Logic - 3 attempts with exponential backoff (3s, 2s delays), drops after failures
  • Draft Auto-save - Progress saved locally with expiration warning
  • Skeleton Loaders - Animated placeholders for question cards, progress bars, full screen
  • Error Dialogs - Network, timeout, disabled evaluation, auth error dialogs with recovery options
  • Responsive Design - Adaptive layout for small (< 400px), medium (400-600px), large (> 600px) screens
